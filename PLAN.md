## Phase 2: Model Training
**Goal:** Trained YOLO + RL regressor models saved to `models/`.

### Step 2A: YOLO Fine-tuning
```bash
# Quick validation run first (5 epochs)
python3 train_yolo.py --epochs 5 --batch 16 --img-size 640 --fast

# Full training (if time permits on Jetson, else skip to pretrained)
python3 train_yolo.py --epochs 50 --batch 16 --img-size 640
```

**Fallback if training time is tight:** Use base `yolov8m.pt` pretrained weights directly in detector.py and skip fine-tuning. The base YOLOv8 can already detect many road elements. Document this as "ready for domain fine-tuning" in README.

### Step 2B: RL Regressor Training
```bash
# Quick validation (1 epoch)
python3 train_rl_model.py --epochs 1 --batch 8 --fast

# Full training
python3 train_rl_model.py --epochs 50 --batch 32
```

**Training targets:**
- RL MAE < 50 mcd/m²/lx (on synthetic labels, this is achievable)
- Qd MAE < 0.05
- Training time: ~30 min on Jetson Xavier NX GPU for 50 epochs

### Step 2C: TRT Conversion (optional, do last)
```bash
python3 convert_trt.py --fp16
```

## Risk Mitigation

| Risk | Probability | Mitigation |
|------|-------------|------------|
| YOLOv8 training too slow on Jetson | Medium | Use pretrained weights + label "ready for fine-tuning" |
| Kaggle dataset class label mismatch | Medium | Hardcode remapping tables per dataset in dataset_preparation.py |
| FLIR Lepton driver issues | High | Mark as optional, always fallback gracefully |
| TensorRT conversion fails | Medium | Keep PyTorch path as default, TRT as optional boost |
| DHT11/IMU not available on dev machine | High | Fallback already required — test fallback path first |
| Streamlit and inference in same process | Low | Use separate processes / shared state via file or queue |

## Innovation Points to Highlight (for 30-mark Innovation score)
Claude: When writing README.md and any documentation, emphasize these unique contributions:

1. **Multi-sensor IoT fusion** — DHT11 wet correction + IMU tilt correction applied to RL calculation (not just measurement)
2. **Thermal night assist** — FLIR Lepton for low-light condition detection (rare in highway systems)
3. **EfficientNet regression model** — not just detection, but quantitative RL/Qd prediction per object
4. **30-metre geometry correction** — mathematically derived from IRC:35-2015 standard, not hardcoded
5. **Moving vehicle measurement** — mobile platform vs. traditional handheld device (addresses NHAI's stated pain point directly)
6. **Drone-compatible architecture** — the same pipeline works for drone-mounted cameras (mention in README)
7. **Real-time GPS-tagged reporting** — map + CSV output that NHAI field engineers can use directly
8. **IRC threshold compliance check** — automated Red/Amber/Green classification against Indian standards
